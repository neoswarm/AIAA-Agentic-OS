**Skill execution form (skill_execute.html):**

The existing `validateForm()` function (line ~691) already adds `has-error` class and the template has `.field-error` elements with generic "This field is required" text. Enhance it:

1. In `validateForm()`, update each `.field-error` span's textContent to be field-specific: "Company name is required", "Blog topic is required", etc. Use the label text to derive the message: `fieldLabel + ' is required'` where fieldLabel comes from the `.field-label` text in the same `.field-group`.

2. Add real-time validation: after `buildForm()` completes, attach `blur` event listeners to all required inputs. On blur, if the field is empty, add `has-error` class and show the error. On `input`, if the field now has a value, remove `has-error` class.

3. For fields with type `url`, add URL format validation on blur: check if value matches a basic URL pattern (`/^https?:\/\/.+/`). Show "Please enter a valid URL (e.g. https://example.com)" error.

4. For fields with type `number`, validate min/max constraints on blur if the input has min/max attributes. Show "Value must be between X and Y" error.

5. Add CSS for `.field-error` animation: add `transition: opacity 0.2s` so errors fade in smoothly rather than appearing abruptly.

**Client form (clients.html):**

1. Add a `<span class="field-error" id="clientNameError"></span>` element after the `#clientName` input field.

2. Add a `<span class="field-error" id="clientWebsiteError"></span>` element after the `#clientWebsite` input field.

3. Add CSS for `.field-error` in the `<style>` block:
```css
.field-error {
    display: none;
    margin-top: 0.25rem;
    font-size: 0.75rem;
    color: var(--error);
}
.field-error.visible {
    display: block;
}
.form-input.has-error,
.form-select.has-error,
.form-textarea.has-error {
    border-color: var(--error);
}
```

4. In the `submitClientForm()` function (line ~479), BEFORE the fetch call, add validation:
   - If `clientName` is empty: show "Company name is required", add `.has-error` to input, return false
   - If `clientWebsite` has a value but doesn't match `/^https?:\/\/.+\..+/`: show "Please enter a valid URL (e.g. https://example.com)", add `.has-error` to input, return false

5. Add `blur` listeners to `#clientName` and `#clientWebsite`:
   - On `#clientName` blur: if empty, show error; on input, if non-empty, clear error
   - On `#clientWebsite` blur: if non-empty and invalid URL format, show error; on input, clear error

6. Create a helper function `showFieldError(inputId, errorId, message)` and `clearFieldError(inputId, errorId)` to DRY the show/hide logic.

**Important:** Do NOT replace the existing `form.checkValidity()` / `form.reportValidity()` browser behavior -- add the inline errors ON TOP of it. The inline errors provide a better UX but the browser validation is a safety net.
  </action>
  <verify>
    1. Open skill_execute.html in browser, leave a required field empty, click "Run Skill" -- red inline error appears under the field
    2. Type into that field -- error disappears
    3. Open clients.html, click "Save Client" with empty name -- inline error appears under Company Name
    4. Enter "not-a-url" in website field, blur -- URL format error appears
    5. Fix the URL to "https://example.com", blur -- error disappears
    6. Grep for 'has-error' in both files to confirm error state toggling is implemented
  </verify>
  <done>
    Skill execution required fields show inline "X is required" on submit and on blur. Client form shows "Company name is required" and URL format validation. Errors clear when user corrects input.
  </done>

<task type="auto">
  <name>Task 2: Add API key format validation to settings page</name>
  <files>
    railway_apps/aiaa_dashboard/templates/settings.html

  <action>
The settings page (settings.html) has inline JS with a `saveApiKey()` function that only checks `if (!value) return`. The server already has `_KEY_PREFIXES` in `api_v2.py`. Add client-side prefix validation BEFORE the save request is sent.

1. Add a `_KEY_PREFIXES` constant object in the inline `<script>` block, mirroring the server-side prefixes:
```javascript
var KEY_PREFIXES = {
    'openrouter': { prefix: 'sk-or-', label: 'OpenRouter API Key', hint: 'Should start with sk-or-' },
    'perplexity': { prefix: 'pplx-', label: 'Perplexity API Key', hint: 'Should start with pplx-' },
    'anthropic':  { prefix: 'sk-ant-', label: 'Anthropic API Key', hint: 'Should start with sk-ant-' },
    'openai':     { prefix: 'sk-', label: 'OpenAI API Key', hint: 'Should start with sk-' },
    'slack':      { prefix: 'https://hooks.slack.com/', label: 'Slack Webhook URL', hint: 'Should start with https://hooks.slack.com/' },
    'fal':        { prefix: '', label: 'fal.ai Key', hint: '' }
};
```

2. Add a `<span class="key-format-error" id="error-{keyName}"></span>` element after each `.api-key-input-row` div for openrouter, perplexity, slack (the ones with prefixes). Add CSS:
```css
.key-format-error {
    display: none;
    margin-top: 0.5rem;
    font-size: 0.75rem;
    color: var(--error);
}
.key-format-error.visible {
    display: block;
}
```

3. Modify `saveApiKey(keyName)` to:
   - Check if value is empty -- show "Please enter an API key" in the error span
   - Check if the prefix is defined AND the value doesn't start with it -- show the hint message (e.g., "OpenRouter key should start with sk-or-")
   - If validation fails, do NOT send the fetch request; show the error and return
   - If validation passes, clear any previous error and proceed with fetch

4. Add `input` event listeners to each key input to clear the error message when the user starts typing (remove `.visible` class from error span).

5. Also handle the server error response: if the fetch returns an error about format, display it in the same error span rather than only as a toast. Parse the response JSON and check for `message` containing "Invalid key format".

**Important:** The "fal" key has no prefix requirement -- skip prefix validation for it. The "google" key has no input field (server-configured) -- skip it entirely.
  </action>
  <verify>
    1. Open settings.html, type "bad-key" into OpenRouter input, click Save -- error "OpenRouter key should start with sk-or-" appears inline, no network request sent
    2. Type "sk-or-valid-key-here" -- error clears on input
    3. Click Save again -- request is sent (no inline error)
    4. Check fal.ai key field has no prefix validation
    5. Grep for 'key-format-error' in settings.html to confirm error elements exist
  </verify>
  <done>
    Settings page validates API key format (prefix) on the client side before sending save request. Error messages are inline under the input field. Keys with no prefix requirement (fal) skip validation. Error messages clear when user edits input.
  </done>
</task>
